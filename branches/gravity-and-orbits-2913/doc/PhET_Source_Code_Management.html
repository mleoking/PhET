PhET Source Code Management

Source code repository

PhET uses SVN (aka, Subversion) for source code control, and the SVN repository is hosted by SourceForge. Anyone can read from the repository. But to commit changes, you'll need a SourceForge login. Ask one of the PhET developers to set you up.

PhET's project page on SourceForge is: http://sourceforge.net/projects/phet/

For configuring SVN clients, the SVN repository URL is: https://phet.svn.sourceforge.net/svnroot/phet

SVN Clients

If you're using Eclipse or IntelliJ IDEA, read on. If you're using something else, you're on your own and can skip this section.

IntelliJ IDEA has integrated SVN support. Consult the IDEA user guide for details.

As of version 3.3, Eclipse does not have built-in SVN support. You'll need to install an SVN client plug-in. Your options are Subversive and Subclipse. We recommend that you use Subversive.

Subversive was formerly a third-party plug-in, but has recently become an official part of the Eclipse project. It is currently in the incubation phase of integration with Eclipse, but will eventually be bundled with Eclipse. Its interface is similar to Eclipse CVS, and it has good performance. Subversive requires Eclipse 3.3 or higher. Get Subversive at http://www.eclipse.org/subversive/. Beware that the installation instructions are in many cases incorrect. If you encounter problems, ask one of the PhET developers for help.

Subclipse is your other option. It is available at http://subclipse.tigris.org/. If you're familiar with Eclipse CVS, you'll find Subclipse to be maddening; it doesn't work the same way. The biggest downside of Subclipse is performance; it can take minutes to synchronize your working copy with the repository. And since Subversive is on track to becoming an integrated part of Eclipse, there is really no reason to use Subclipse.

Repository Organization

The repository organization is documented in these README files:

  • trunk/README.txt : top-level directories
  • trunk/simulations-java/README.txt : all Java simulations
  • trunk/simulations-java/simulations/README.txt : individual Java simulations

TERMINOLOGY ALERT! Each directory under trunk/simulations-java/simulations is referred to as a project, and contains the code for one or more simulations. Each simulation built from a project is referred to as a flavor. Some projects may have only 1 flavor, or may have a flavor name that is identical to the project name.

Checking out source code for Java simulations

All PhET source code is under trunk; we never create branches or tags.

All Java simulations live under trunk/simulations-java.

If you're using Eclipse or IntelliJ IDEA, read on. If you're using something else, you're on your own and can skip the remainder of this section.

If you're using Eclipse, check out trunk/simulations-java as a Java project named simulations-java. Then follow the instructions in simulations-java/ide/eclipse/README.txt.

If you're using IntelliJ IDEA, look in simulations-java/ide/intellij for similar instructions, or contact Sam Reid.

Test your working copy by running class FaradayApplication as a Java application. If you experience build errors, then your classpath is set incorrectly. Re-read the above instructions, then ask a PhET developer for help.

Building and deploying simulations

Ant is used to build and deploy simulations. See simulations-java/build.xml and build-ancillary.xml for details. The most common targets are documented at the top of build.xml.

If you are not a member of the PhET development team, it is unlikely that you will be able to use any of the build targets with deploy in their name. These targets deploy simulations to the PhET productions and development servers; they require logins for the PhET servers, as well as commit permissions for the SVN repository.

WARNING: Until you are very familiar with how simulations are built and deployed, please use only the launch-sims (default) target.

NOTE: Because the buildfiles make heavy use of Antform, you may not be able to run builds from inside Eclipse. This is due to a general problem with AWT threads. If you experience this problem, use the build.sh (Mac, Linux) or build.bat (Windows) scripts to run builds.
